## DNS Records To Create

### Shared Postal records

Create the shared Postal records generated by the Postal installation:

- A/AAAA for the UI/API host
- MX records for inbound/return-path handling
- SPF include host
- return-path host
- tracking host if click/open tracking will be enabled later

### Per-product sending-domain records

For each `mail.<product-domain>` add the records Postal generates for that domain/server:

- SPF TXT
- DKIM record(s)
- return-path or routed host record(s)
- tracking CNAME if enabled

## Sample Pattern

The exact values should come from Postal’s domain/server UI, but the target pattern should look like:

- `mail.fuel.codyjo.com TXT "v=spf1 include:spf.postal.codyjo.com ~all"`
- `postal._domainkey.mail.fuel.codyjo.com` or the DKIM selector Postal provides
- `rp.mail.fuel.codyjo.com` or the routed return-path host Postal provides

Repeat the same model for Study, Selah, Tarot, Cordivent, and Cody Jo Method.

## Secrets / Environment Actions

Each product deployment needs:

- `EMAIL_PROVIDER=postal`
- `EMAIL_POSTAL_BASE_URL=https://postal.codyjo.com`
- `EMAIL_POSTAL_SERVER_KEY=<product-specific-server-key>`
- `EMAIL_FROM_NAME=<product-name>`
- `EMAIL_FROM_ADDRESS=no-reply@mail.<product-domain>`
- `EMAIL_REPLY_TO=<support address for that product>`

## AWS / CI Actions Required

- Add the Postal server keys to the deployment secret source used by each repo.
- Ensure each CodeBuild/CD path injects the new `TF_VAR_email_*` variables or equivalent runtime envs.
- Deploy Terraform/env changes per repo after the DNS records are live.

## Validation Steps

For each product:

1. Verify Postal shows the sending domain as valid.
2. Deploy with `EMAIL_PROVIDER=postal`.
3. Trigger:
   - registration verification
   - resend verification
   - forgot-password
   - reset-password
4. Confirm the message headers show the correct From domain.
5. Confirm SPF/DKIM alignment and inbox placement in at least Gmail + Outlook.
6. Watch Postal logs for rate limiting, rejects, or DNS validation failures.
